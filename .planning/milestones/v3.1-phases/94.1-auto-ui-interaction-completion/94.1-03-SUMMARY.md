# Phase 94.1 Plan 03: Board-Bridge Checkers Destination Fix Summary

**One-liner:** Fixed two bugs in useBoardActionBridge — board.clear() timing (wiped board after watcher populated it) and selection.choices guard (blocked dynamic choice branch) — making Checkers destination picks clickable on the board after piece selection.

## What Was Built

Proved and fixed Bug A: Checkers board-select of a piece did not surface destination choices as selectable board cells after the piece was clicked.

**Root cause 1 (Rule 1 deviation found during investigation):** `board.clear()` in `startAction` was placed AFTER `await controller.start()`. The async fetch inside `controller.start()` triggers `snapshotVersion++` which runs watcher D (board population) as a microtask BEFORE `startAction` resumes. So `board.clear()` ran after the watcher had already set `board.validElements`, wiping them. Watcher D's sources (currentPick, filteredValidElements) were unchanged after the clear, so it never re-ran. Board stayed empty.

**Fix 1 (Rule 1, in Task 1):** Move `board.clear()` BEFORE `await controller.start()`. At call time, the board is already clean (auto-start requires `currentAction === null`, and watcher E already called `board.clear()` when the previous action ended). The comment in startAction explains the ordering requirement.

**Root cause 2 (Bug A, fixed in Task 2):** Watcher D's choice-pick branch had condition `selection.type === 'choice' && selection.choices`. `selection.choices` is a static metadata field — it is `undefined` for `chooseFrom` calls that use `fetchPickChoices` (dynamic fetching). The condition evaluated false, so the branch never ran for Checkers destination picks. Board received `setValidElements([], () => {})` instead of the fetched destination cells.

**Fix 2 (Task 2):** Remove `&& selection.choices` guard. The actual choices are in `choicesForBoard.value` (reactive, reads `snapshotVersion`), which is already used inside the branch. Added comment explaining why `selection.choices` is not the source of truth for dynamic choices.

**Fix 3 (Task 2, same commit):** `setSelectionValue` had the same `selection.choices` guard for post-fill visual highlighting. Replaced with `choicesForBoard.value.slice()` snapshot captured BEFORE `controller.fill()` — fill() advances the pick state, so the snapshot taken afterward would show the next step's choices.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] board.clear() timing in startAction wipes watcher D's board population**

- **Found during:** Task 1 (RED test investigation — A1 failed at wrong assertion `id:42` not selectable instead of `notation:a5`)
- **Issue:** In `startAction`, `board.clear()` was called AFTER `await controller.start()`. The async fetch completes in the microtask queue before `startAction` resumes. Specifically: `snapshotVersion++` → `queueFlush()` schedules the Vue scheduler flush as a microtask → watcher D fires → `board.setValidElements([piece_or_dest])`. Then `startAction` resumes (a few hops later) and calls `board.clear()`, wiping the validElements. Watcher D's sources (currentPick, filteredValidElements, choicesForBoard) were unchanged by `board.clear()`, so it never re-ran. Confirmed by adding a `watch(() => board.validElements.length, ...)` tracer.
- **Fix:** Move `board.clear()` to BEFORE `await controller.start()`. Remove the now-redundant `board.setCurrentAction(actionName, 0, firstSel.name)` after the await — watcher E (immediate, watches `controller.currentAction`) fires during the async chain and calls `board.setCurrentAction` with the accurate pick index from `currentPick.value`.
- **Files modified:** `src/ui/composables/useBoardActionBridge.ts`
- **Commit:** 3bb4626 (included in Task 1 RED commit as a prerequisite for the test to fail at the correct assertion)
