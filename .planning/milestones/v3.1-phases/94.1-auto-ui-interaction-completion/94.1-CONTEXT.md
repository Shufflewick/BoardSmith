<decisions>
## Implementation Decisions

### Test infrastructure FIRST (locked by user)
- **D-01:** Before fixing interaction, stand up component/interaction test infra (likely
  `@vue/test-utils` + jsdom/happy-dom — DISCUSS the dependency with the user first). The
  acceptance bar for the interaction fixes is that they're covered by component-level tests that
  reproduce the browser failures below — not just controller-unit tests (which passed while the
  browser failed). This is the structural mechanism that prevents the non-convergence Phase 94 hit.

### Footer ActionPanel = reliable fallback (strategy locked by user)
- **D-02:** Suppress the footer ONLY when the board can FULLY drive the current step — i.e. it
  currently offers a clickable board element for every current choice. Otherwise the footer stays
  present with **populated** choices. Board-centric clicking is a progressive enhancement layered
  on top where it works (Hex; Checkers piece-select). **Proposed mechanism (verified to flip the
  footer on, but insufficient alone — see Bug C):** change `allCurrentChoicesAnchored` in
  `src/ui/composables/useActionController.ts` to `return validElements.value.length > 0;`
  (replacing the per-choice notation predicate). This was tried, reverted to keep `main` green
  because it breaks 5 round-2 tests that assert the OLD semantics (`useActionController.test.ts`
  ~lines 1987, 2060 — "becomes anchored reactively…Checkers destination step"). Those tests must
  be UPDATED to the new semantics as part of this phase.

### Claude's Discretion
- Exact test-infra choice/config, how board-select of a multi-step source triggers the next
  step's choice fetch+population, and renderer affordance details — planner's discretion, but
  every fix must be reproduced by a component test first.
</decisions>

## REMAINING bugs (reproduce these in component tests, then fix)
- **Bug A — Checkers move can't complete.** Select a red piece (board click highlights its square);
  the destination `chooseFrom` step does NOT surface selectable destination cells, and (pre-footer-fix)
  the footer was suppressed → stuck. Clicking a destination cell only highlights it (`selectElement`),
  never executes.
- **Bug B — Go Fish "ask" can't complete.** On the player's turn, 0 `.action-selectable`, footer
  suppressed; the ask (target player + rank) surfaces no affordance and no panel.
- **Bug C — choice population (the deepest).** With the D-02 footer-fallback applied, the footer
  correctly REAPPEARS for the Checkers destination step ("Move | b6 | Select destination"), but shows
  "**No options available**" even though the piece has legal moves (A5/C5). So board-selecting the
  multi-step SOURCE advances the pick but the next step's filtered `chooseFrom` choices never
  fetch/populate (in panel OR board). Likely: board-select via the bridge doesn't trigger the
  `fetchChoicesForPick` that the panel-driven flow does, and/or the `filterBy:{key:'pieceId',
  selectionName:'piece'}` filter doesn't match the board-selected piece arg. THIS is the crux —
  the footer fallback (D-02) is necessary but not sufficient without fixing choice population.
