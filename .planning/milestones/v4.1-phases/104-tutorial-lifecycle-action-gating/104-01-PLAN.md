Serialization rules (src/engine/element/game-element.ts:699-714, 820-853):
- `toJSON()` enumerates `Object.keys(this)`, skips keys in `unserializableAttributes` and any
  key starting with `_`. A `#private` field or getter is INVISIBLE to this loop -> silent data loss.
- `serializeValue` encodes a `Map` losslessly as `{ __map: [[k,v],...] }` (numeric keys preserved);
  a plain object stringifies its keys. Use `Map<number, TutorialProgress>`.

Restore (src/engine/element/game.ts:2483-2487): `loadSerializedState` re-applies own attributes by
direct assignment for every key not in `unserializableAttributes`, not `_`-prefixed, not handled.

unserializableAttributes (src/engine/element/game.ts:445-460): array currently lists
`'pile','random','commandHistory','_actions','_actionExecutor','_flowDefinition','_flowEngine',
'_debugRegistry','_constructorOptions'`. Add `tutorialDefinition` here (definition is static config,
must NOT serialize — mirrors `_actions`).

GameOptions (src/engine/element/game.ts:145-162): constructor param. `_constructorOptions` stores a
spread copy for restore. GameRunner builds the game via `new options.GameClass(options.gameOptions)`
(src/runtime/runner.ts:94) and on restore via `fromSnapshot` rebuilds `gameOptions` from
`snapshot.gameOptions` (src/runtime/runner.ts:357-368).

GameDefinition (src/session/types.ts:64-78): already carries optional `ai`, `gameOptions`,
`playerOptions`, `presets`. `tutorial` is a natural peer.

PlayerGameState (src/session/types.ts:360-386): the client projection. Reserve two optional slots
here now (populated in Plan 104-04): `tutorial?` (active step id + reserved content) and
`disabledActions?: Record<string,string>` (action-name -> gate reason).

AIController precedent (src/session/game-session.ts:565-566): `GameDefinition.ai` is passed
un-serialized into a controller. Mirror this threading style for the tutorial definition.
