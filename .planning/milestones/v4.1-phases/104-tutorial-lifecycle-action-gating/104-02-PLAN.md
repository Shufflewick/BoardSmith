<objective>
Implement the tutorial action/target gate by AUGMENTING the engine's existing v2.8 disabled path —
no parallel validator (success criterion #4). Target/choice gating ORs a gate reason into
`getChoices`' `AnnotatedChoice.disabled`; whole-action gating is computed as a separate action-name ->
reason map (because action availability is a binary `string[]` with no action-level disabled on the wire,
RESEARCH finding #3 / Pitfall 3).

Purpose: Deliver TUT-02 enforcement entirely inside the engine so every downstream consumer (validation,
availability, UI metadata, pick-handler) inherits the gate for free.
Output: `src/engine/tutorial/gate.ts`, a `getChoices` augmentation, a `Game` action-level reason method,
and the engine gate test.
</objective>

<interfaces>
<!-- Extracted from codebase. Use directly. -->

getChoices (src/engine/action/action.ts:316-441): returns `AnnotatedChoice<unknown>[]` for each
selection type. Each branch already computes `disabled: sel.disabled ? sel.disabled(value, context) : false`.
The gate must OR-in: if the active step gates this selection's value out for `context.player.seat`,
set `disabled` to the gate reason string (a non-empty string when gated, keep the existing reason
otherwise). NEVER disable ALL choices of a required selection — that makes the action vanish
(hasValidSelectionPath returns false at action.ts:1218-1221). Always leave the intended target enabled.

validateSelection (src/engine/action/action.ts:677-680, elements variant :704-709): already checks
disabled FIRST and emits `Selection disabled: <reason>`. Reusing `disabled` means a learner submitting a
non-allowed target gets the reason for free.

isActionAvailable / hasValidSelectionPath (src/engine/action/action.ts:947-960, 1121-1245): availability
is binary; `hasValidSelectionPath` keeps the action if >=1 enabled choice exists per required selection.

getAvailableActions (src/engine/element/game.ts:837-845): returns `ActionDefinition[]` (binary). Do NOT
remove gated actions here. Instead add a sibling method (e.g. `getTutorialDisabledActions(seat): Record<string,string>`)
that returns, for the active step, the reason each currently-available action is NOT part of the step.
An action is "gated out" when the step's allow-list does not include it.

AnnotatedChoice (src/engine/action/types.ts:15-23): `{ value, disabled: string | false }`. No bare `true`.

Game tutorial state (from Plan 104-01): `game.tutorialProgress: Map<number, TutorialProgress>`,
`game.tutorialDefinition?: TutorialDefinition`. The active step for a seat = look up progress.stepId in
the definition's steps. Gate is active ONLY when progress.status === 'running' for that seat.
