<objective>
Establish the structural-parity substrate: a single shared anchor attribute set emitted ONCE
in the shared selectable/board layer so the overlay can locate any target without any
renderer or custom UI implementing a new method. Per 105-CONTEXT Area 2 + 105-UI-SPEC §5:

- `anchorAttrs(ref: ElementRef)` in `useBoardInteraction.ts` maps an identity to
  `{ 'data-bs-el-id', 'data-bs-el-notation', 'data-bs-el-name' }` (only the keys present on the
  ref), mirroring `matchesRef` precedence — the SAME parity contract used for highlight/select.
- `useSelectable.attrs` merges `anchorAttrs(identity())` so the 5 element-mode renderers that
  spread the full attrs object (Card, Piece, Die, Deck, Space) get anchors for free.
  **HandRenderer is the exception** — its root `.hand-container` cherry-picks only
  `:tabindex`/`:aria-disabled` from `selectableAttrs` (HandRenderer.vue:275-276) and carries a
  renderer-specific `:data-zone-id`, so it does NOT inherit the merged anchor. It MUST be edited
  explicitly (Task 2) or the Hand primitive gets annotations in a custom UI but not in AutoUI —
  violating the "no primitive in only one UI path" hard rule.
- `useSelectableGrid` returns a `cellAttrs(cell)` helper (calling the same `anchorAttrs`) so the
  2 grid renderers (Grid, Hex) emit anchors on each `.grid-cell` / hex cell.
- Action controls get `:data-bs-action="action.name"`; the ActionPanel root gets `data-bs-panel`.

Purpose: This is the load-bearing hard-rule mechanism. Because the anchor is set in ONE shared
place, a custom UI that uses the shared interaction layer CANNOT "forget" annotation support —
parity is structural, not opt-in. No `isAnnotationHighlighted()` is added anywhere.
Output: data-bs-* anchors on every interactive target + a test proving the single source.
</objective>
