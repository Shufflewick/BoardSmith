<objective>
Prove the hard rule (criterion #3, and the project's "no primitive may work in only one UI path"):
the SAME annotation from the SAME `step.content` renders identically in BOTH UI paths because
targeting routes through the shared anchor layer — NOT through any per-renderer method.

Per 105-CONTEXT specifics + 105-UI-SPEC §5, this test mounts:
1. An **AutoUI renderer fixture** — a real built-in renderer (e.g. GridBoardRenderer or
   PieceRenderer) wired through `useBoardInteraction`/`useSelectable`, so it emits data-bs-el-*
   anchors via Plan 02 — plus TutorialOverlay reading the same step.content.
2. A **custom-UI fixture** — a minimal hand-rolled board that uses the shared selectable layer
   (the same `anchorAttrs` / `useSelectable` path a real custom UI uses) and renders a target with
   the matching anchor — plus the same TutorialOverlay + same step.content.

Both must resolve the same target and render a ring + the same bubble text. This is the
load-bearing acceptance for the entire phase.

Purpose: This is the must_have that proves structural parity is real, not asserted. If a custom
UI could "forget" annotations, this test would expose it.
Output: TutorialOverlay.parity.test.ts — the dual-path proof.
</objective>
