<objective>
Build the AI-vs-AI narrated demo loop in `SnapshotSessionHost` (the only tier with the broadcast adapter + a cancellable async lifetime). `demo-start` and `demo-stop` are host-level lifecycle ops handled directly in `handleOp` (NOT in executeOp). Each loop iteration: run a new read-only `aiSuggest` op to PREVIEW the move, inject narration into transient state and broadcast it BEFORE execution (Phase 107 onBeforeMove semantics), wait the configurable delay, then execute the EXACT same move via the existing `action` op — never re-running MCTS for execution (avoids the narrate/execute mismatch). The loop is guarded by a `demoAbort` flag checked before AND after the delay, a `MAX_DEMO_MOVES` cap, and a `finally` cleanup that always clears state and broadcasts `isDemoRunning=false`.

Purpose: The third and riskiest AI teaching beat. The CLAUDE.md hard rule — never leave a timer/process running — is the dominant constraint; tests must assert no pending timer survives stop or game-over.
Output: aiSuggest op + runDemoLoop + demo-start/demo-stop wiring + fake-timer tests proving clean cancellation.
</objective>
