<objective>
PIT-02: During the FIRST `startFlow()` traversal, record every element class queried through the `GameElement` finder methods (`all`/`first`/`firstN`/`last`/`lastN`), then diff the recorded set against the class registry. Throw, naming the class and steering to `game.registerElements([Class])`, on the first class that was queried but never registered. Post-start / async queries are out of scope (documented limitation). Recording is per-game (held on `_ctx`) and active only during the first traversal, so it imposes no ongoing overhead and never leaks across games/tests.

Purpose: A class queried via `game.all(Foo)` but never registered (typo or dead class reference) currently misbehaves silently — and because an unregistered query returns an EMPTY collection, the check must happen at the query call site (where the class arg is available), not inside the resulting collection.
Output: Recording of class-typed finder args in the 5 `GameElement` finder methods, a first-traversal recording window in `startFlow()`, and PIT-02 unit tests.
</objective>

Corrected mechanism (verified 2026-07-01 — supersedes RESEARCH's "_finder choke point" suggestion, which is NOT viable):

WHY NOT ElementCollection._finder(): `ElementCollection extends Array<T>` has NO `_ctx`, NO `game` field, NO context-bearing constructor (`grep _ctx element-collection.ts` → 0 hits). The GameElement finder methods build a BARE `new ElementCollection(...this._t.children)` and delegate to `_finder`, so `_finder` has no path back to the game/registry. Worse: an UNREGISTERED-class query returns an EMPTY collection, so there is no element inside to reach context from — and the empty case is exactly what PIT-02 must catch. The hook MUST live at the finder call site, not the collection.

CORRECT HOOK — the 5 GameElement finder methods (src/engine/element/game-element.ts, verified):
- all() at ~411, first() at ~446, firstN() at ~477, last() at ~510, lastN() at ~538. Each has the overload `<F>(classNameOrFinder?: ElementClass<F> | ElementFinder<F>, ...finders)` and body `const collection = new ElementCollection(...this._t.children); return collection.<m>(classNameOrFinder as ElementClass<F>, ...finders);`. All run with `this._ctx` in scope.
- The queried class arrives as the first arg when it is a class constructor. Reuse the existing `isElementClass`-style guard (the same guard element-collection.ts uses at l417) to detect "arg is a class, not a filter" before recording. Record `classNameOrFinder` (the ElementClass) — that captures the class even when the query result is empty.

Registry + recording state (src/engine/element/game-element.ts):
- `_ctx!: ElementContext` (l82); `_ctx.classRegistry: Map<string, ElementClass>` (initialized l111-112, populated on element creation l321-322 via `set(className, elementClass)`). `_ctx` is SHARED across the whole element tree of one game, so it is the correct per-game home for both the recorded-classes Set and the "recording active" flag. Add these two fields to the ElementContext type (follow the `ElementContext` import in game-element.ts to its definition) as optional internal fields; do NOT expose them as public API.
- Registry source of truth to diff against: `this._ctx.classRegistry` keys are class names. `registerElements([...])` (game.ts:744, protected) and `createElement()` (auto-register l767-769) populate it. A class is "used but unregistered" only when queried but never instantiated/registered (typo / dead reference) — the positive test must query a class whose elements were created normally (→ registered → no throw).

startFlow() hook (src/engine/element/game.ts:1554): body is `if (!this._flowDefinition) throw; [plan-02 #validateActionReachability() call]; this._flowEngine = new FlowEngine(...); const state = this._flowEngine.start(); ...`. PIT-02's recording must wrap the ACTUAL first execution pass: set `_ctx` recording-active ON, call `start()` inside try/finally, turn recording OFF in finally, then run the diff. Do NOT disturb the plan-02 `#validateActionReachability()` call that precedes `new FlowEngine(...)`.
- Question resolved (RESEARCH Open Q1): FlowDefinition.setup() runs inside engine.start(), so its class queries are captured naturally.
