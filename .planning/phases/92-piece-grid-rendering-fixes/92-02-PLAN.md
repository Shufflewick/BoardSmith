## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| game-author definition → DOM | `$images.face` URLs/sprite paths and element names/ids authored in the game definition are rendered into `<img src>`, CSS `background-image: url(...)`, and text nodes. |
| serialized (visibility-filtered) game view → render | The board renders only the already-filtered view; this phase adds no path that reaches across visibility. Player color is read from the serialized `attributes.player.color` that the engine already exposes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-92-01 | Tampering (DOM injection / XSS) | piece-image `:src`, piece-token/label, error-panel text | mitigate | All author strings render via Vue attribute bindings and `{{ }}` text interpolation (auto-escaped); `v-html` is forbidden and asserted absent. `<img src>` does not execute `javascript:` URIs. |
| T-92-02 | Tampering (CSS injection) | piece-sprite `background-image: url(...)` | accept | The sprite URL is game-author-controlled (same trust level as code), never end-user/opponent input; it cannot ride in from another player's serialized view. Risk is authoring self-harm only. |
| T-92-03 | Information Disclosure | grid error panel + console.error | mitigate | Error text is the fixed D-03 literal (element id + prop name only) — no file paths, line numbers, or stack fragments. UI-SPEC copywriting rules forbid internal identifiers. |
