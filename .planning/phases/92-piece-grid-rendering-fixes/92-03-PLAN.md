<objective>
Confirm in the browser that the helper-driven rendering changes from Plan 02 produce the
correct visuals end-to-end: image pieces, owner-colored token pieces, a correctly-sized
declared grid, and a loud in-board error panel for a misconfigured grid. This is the single
manual verification called out in 92-VALIDATION.md — pure helper logic is already covered by
automated tests (Plan 01); only live DOM rendering requires the eye.

Purpose: Per project CLAUDE.md, features are verified by running the application in the
browser, not just by green unit tests. This checkpoint closes that gap for PIECE-01/02/03.
Output: Human confirmation (or a logged defect list to feed gap closure).

Integration-test boundary (explicit decision, not a gap): CLAUDE.md asks for an automated
integration test per cross-layer boundary. The boundary this phase touches is engine
serialization → AutoElement.vue rendering. By established project pattern, that boundary is
NOT covered by an automated SFC test: the project has no Vue component test infrastructure
(confirmed by 92-RESEARCH.md), and adding one (e.g. Vue Test Utils) would itself require a
dependency discussion (CLAUDE.md: "Don't add dependencies without discussing"). Instead,
Plan 01 extracts all cross-layer logic into pure, unit-tested helpers (`resolvePieceVisual`,
`resolveGridSize`), and THIS plan's browser verification serves as the integration gate for
the engine→Vue rendering boundary — exercising the full engine→UI stack with a real game.
Do NOT add a Vue test dependency to satisfy this; the human browser check is the agreed gate.
</objective>
