## Archetype Template Visual Hierarchy

This section specifies the three archetype templates selected by `selectArchetype()`. Each encodes visual hierarchy — NOT equal-space subdivision. The C6 failure mode (Boardzilla's democratic equal-space subdivision) is explicitly prohibited.

### Template 1: GridBoardTemplate (grid-board archetype)

**When selected:** `$layout === 'grid'` or `$layout === 'hex-grid'` present anywhere in top-level children.

**Layout structure:**

```
┌─────────────────────────────────────────────┐
│  [peripheral chrome: player info, scores]   │  auto height (~44px)
├─────────────────────────────────────────────┤
│                                             │
│           FOCAL BOARD                       │  flex: 1 (dominant area)
│      (GridBoardRenderer or                  │
│       HexBoardRenderer)                     │
│                                             │
├─────────────────────────────────────────────┤
│  [docked hand strip — present only if       │  auto height (max 30vh)
│   hand elements exist]                      │
└─────────────────────────────────────────────┘
```

**CSS prescription:**
```css
.grid-board-template {
  display: grid;
  grid-template-rows: auto 1fr auto;
  grid-template-areas:
    "chrome"
    "board"
    "hand";
  height: 100%;
  overflow: hidden;
}
.grid-board-template__chrome  { grid-area: chrome; }
.grid-board-template__board   { grid-area: board; overflow: auto; }
.grid-board-template__hand    { grid-area: hand; max-height: 30vh; overflow-x: auto; }
```

The board area (`1fr`) takes all remaining height. The hand strip is `auto` — absent entirely if no hand elements exist. The chrome slot is `auto` — collapsed if no peripheral elements. The board is the visually dominant element at all viewport sizes.

**Non-grid peripheral elements** (decks, discard piles that are not the focal board) are placed in the chrome slot or inlined above the board — never given equal grid area alongside the board.

### Template 2: CardTemplate (card archetype)

**When selected:** No grid/hex board; card/hand/deck elements are ≥50% of top-level children.

**Layout structure:**

```
┌─────────────────────────────────────────────┐
│  [peripheral: deck(s), discard pile(s)]     │  auto height
├─────────────────────────────────────────────┤
│                                             │
│           HAND(S) — dominant                │  flex: 1
│      (HandRenderer centered)                │
│                                             │
└─────────────────────────────────────────────┘
```

**CSS prescription:** `display: flex; flex-direction: column;` — peripheral zones above, hand(s) below filling remaining space. No fixed-size board slot.

### Template 3: TableauTemplate (fallback)

**When selected:** Neither of the above; addressable topologies (stack, hand, space containers) only.

**Layout structure:** Flex-wrap of zone containers. No prescribed hierarchy — this is the general catch-all for addressable-but-unspecialized layouts. Still must never produce a blank screen — every zone renders via its per-element renderer.

### Template 4: UnsupportedTopologyPanel (honest-fail boundary)

**When selected:** Free-form layout (`$layout === 'free-form'`) or topology that yields no addressable elements.

**Design requirements:**
- Full-width, visually prominent panel — not a subtle subtitle
- Amber/warning color treatment on header text (`#f59e0b`)
- White body text on the standard dark surface
- `border-radius: 8px`, `padding: 24px`, `border: 1px solid rgba(245, 158, 11, 0.4)`
- Background: `rgba(245, 158, 11, 0.1)` (same treatment as time-travel-banner in GameShell)
- Never renders a blank screen or a partial degraded layout

## Per-Element Renderer Visual Contracts (Sane Floor)

Each built-in renderer must guarantee a visible, non-blank render with zero game-specific configuration. These are the "never blank" contracts per §2b of the research doc.

**Baseline (sprite sheet):**
- `background-image`, `background-position`, `background-size` set from sprite descriptor; same dimensions and border-radius

**Baseline (no image — sane floor):**

## Animation Contracts

**Phase 93 consumes existing animation infrastructure — it does not rebuild it.**

| Animation | Source | Trigger | Timing |
|-----------|--------|---------|--------|
| Deal (card flies from deck to hand/zone) | `useAnimationEvents` + `useFlyingElements.fly()` | `'deal'` animation event type | `useFlyingElements` default (library controls timing) |
| Flip (card back→face) | `useAnimationEvents` + CSS card-flip from `card-flip.css` | `'flip'` animation event type | `0.6s` (from existing card-flip.css) |
| Reveal (card becomes visible to this player) | `useAnimationEvents` + fly + flip combo | `'reveal'` animation event type | combined fly + flip timings |
| Element move (FLIP) | `useFLIP` composable | game state change (implicit) | `0.3s ease` (FLIP default) |
| Hex piece entrance | CSS `<Transition>` | piece appears in hex cell | `0.15s ease` (opacity + scale) |
| Card hover lift | CSS `:hover` | user hover | `0.2s ease` |
| Cell hover | CSS `:hover` | user hover | `0.15s ease` |

**Animation event wiring pattern:**
- `AutoRenderer.vue` calls `useAnimationEvents()` (inject only — never `createAnimationEvents`)
- If the injection returns `undefined` (GameShell didn't provide it), all handlers are skipped gracefully — no error thrown
- Each handler is defensive: if `event.data` is missing expected keys, the animation silently skips rather than throwing

**Flying cards overlay:** `FlyingCardsOverlay` component (from `src/ui/components/helpers/`) must be rendered inside `AutoRenderer.vue` to serve as the animation canvas for flying element trajectories.

**FLIP:** `AutoRenderer.vue` consumes FLIP via the existing composable. Do not re-implement the `capturePositions`/`animateMovements` dance from `AutoGameBoard.vue` — use `useAnimationEvents` 'deal' events as the semantic trigger instead.

## Phase 93 Carry-Forward Acceptance Requirements

These are requirements carried from Phase 92's UI review that the new renderer MUST satisfy (from `93-UI-CARRYFORWARD.md`). The checker will verify these:

1. **Piece hover affordance:** All non-hex-board piece elements must include `transform: scale(1.05)` on `:hover`. The transition is already present — the hover rule must be explicitly declared.

2. **Single shadow on token pieces:** A piece rendered as a labeled token (no image) must carry exactly one `box-shadow` declaration. No shadow on both a `.piece` wrapper AND a `.piece-token` inner element — the old double-stacking bug must not be reproduced.

3. **Grid error panel copy:** The error panel text must be exactly: `Grid "<id>" can't render — declare $rowCoord/$colCoord on the Grid element` — matching the string in `auto-ui-helpers.ts:169`. Capitalization, slash syntax, and phrasing are locked.
