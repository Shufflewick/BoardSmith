## Interaction Affordance Contract

> **D-03 / INTERACT-01:** All board-interactive elements must express exactly these six
> visual states. Parity between auto-UI renderers and custom UIs is enforced via the
> shared `useBoardInteraction` substrate.

### Six Required Interaction States

| State | CSS class | Visual specification | Trigger |
|-------|-----------|---------------------|---------|
| Default | _(none)_ | Cell/card at rest: `rgba(255,255,255,0.10)` bg | No action in progress |
| Hoverable | `:hover` | `rgba(0,217,255,0.20)` bg — cursor changes to `pointer` only when selectable | Pointer enters cell |
| Selectable | `.action-selectable` | `rgba(46,204,113,0.20)` bg + `2px solid rgba(46,204,113,0.60)` outline (inset) + `pulse` animation | `boardInteraction.isSelectableElement()` is true |
| Selected (source) | `.is-board-selected` | `rgba(0,255,136,0.30)` bg (grid CSS) or fill+stroke `rgba(0,255,136,0.80)` (SVG hex) | `boardInteraction.isSelected()` is true |
| Highlighted (target/multi) | `.is-board-highlighted` | `rgba(0,217,255,0.30)` bg (grid) or fill+stroke `rgba(0,217,255,0.80)` (hex) | `boardInteraction.isHighlighted()` is true — applies for ALL refs roles (`target`, `source`, `highlight`) in the `refs` array |
| Disabled | `.is-disabled` | `opacity: 0.35`, `cursor: not-allowed`, no hover effect | `boardInteraction.isDisabledElement()` is true |

**Role-specific highlight:** When a multi-ref action has entries with `role: 'source'`,
`role: 'target'`, and `role: 'highlight'`, ALL matching cells receive `.is-board-highlighted`
simultaneously. The visual treatment is identical across roles at this phase — role-specific
color differentiation (e.g. source=green, target=cyan) is a future enhancement. The contract
here is: every cell in the `refs` array shows `.is-board-highlighted` at the same time.

**Disabled state spec:** `opacity: 0.35` applied to the element root. `cursor: not-allowed`.
No hover background change. No outline. Guard: `if (isDisabled) return` in click handler.

**Draggable state:** `.is-draggable` → `cursor: grab`. `.is-dragging` → `opacity: 0.50`,
`transform: scale(0.95)`. Source: existing `PieceRenderer.vue` — carry forward unchanged.

### Multi-ref Simultaneous Highlight

A Checkers multi-jump produces a `refs` array like:
```typescript
refs: [
  { ref: { notation: 'e5' }, role: 'source' },
  { ref: { notation: 'c3' }, role: 'target' },
  { ref: { notation: 'd4' }, role: 'highlight' },   // captured square
]
```

ALL three cells receive `.is-board-highlighted` at once. The renderer calls
`boardInteraction.isHighlighted({ id, name })` for each cell independently — the
interaction substrate tracks all highlights simultaneously. No special renderer
logic is needed beyond calling the existing method per cell.

**`matchesRef` is the authoritative ref-matching function** (`useBoardInteraction.ts`).
Renderers must pass `{ id, name }` identity objects — never raw element refs directly.
If `element.notation` is not at the top level of the renderer's element object, extract
it from `element.attributes` and pass it explicitly.

## ActionPanel Suppression Contract

> **D-02 / INTERACT-02:** The footer ActionPanel is `v-if`-absent (not `v-show`-hidden).
> Using `v-show` is a defect — it takes DOM space and obscures board content.

### Auto-Absent Rule

The `allCurrentChoicesAnchored` computed in `useActionController.ts` drives GameShell:

```
currentPick === null            → allCurrentChoicesAnchored = false  → panel SHOWS (player starts actions)
currentPick.type = 'element'    → allCurrentChoicesAnchored = true   → panel ABSENT (always anchored)
currentPick.type = 'elements'   → allCurrentChoicesAnchored = true   → panel ABSENT (always anchored)
currentPick.type = 'choice' + every choice has refs.length > 0 → panel ABSENT
currentPick.type = 'choice' + at least one choice has refs.length = 0 → panel SHOWS (hybrid)
currentPick.type = 'number'     → allCurrentChoicesAnchored = false  → panel SHOWS (no anchor)
currentPick.type = 'text'       → allCurrentChoicesAnchored = false  → panel SHOWS (no anchor)
```

**Critical invariant:** When `currentPick === null` (no action in progress), the panel
must be visible. This is how players start their first action. `allCurrentChoicesAnchored`
MUST return `false` for `null` pick.

### GameShell Footer Structure

```vue
<!-- GameShell.vue — footer at line 1279-1295 (currently unconditional) -->
<!-- Phase 94 change: add v-if at <footer> level, not at <ActionPanel> level -->
<footer
  v-if="!props.suppressActionPanel && !actionController.allCurrentChoicesAnchored.value"
  class="game-shell__action-bar"
>
  <slot name="action-panel">
    <ActionPanel ... />
  </slot>
</footer>
```

**When footer is absent:** No footer chrome takes space. The board occupies the full
height that the footer would have used. No placeholder, no empty div.

**Explicit escape hatches (D-02):**
- `:suppress-action-panel="true"` prop on `<GameShell>` — forces panel absence regardless
  of anchor state. Default: `false`.
- `#action-panel` slot on `<GameShell>` — replaces the entire `<ActionPanel>` with
  custom content while keeping the footer chrome. Use this for hybrid or custom action UIs.

### Hybrid Layout (D-03 / INTERACT-01)

When a `choice` pick has a MIXED set of choices (some anchored, some not):

| Condition | Board | Panel shows |
|-----------|-------|-------------|
| All choices anchored | All choices highlight on board | Panel absent |
| No choices anchored | Nothing highlights | All choices in panel |
| Mixed (some anchored, some not) | Anchored choices highlight on board | ONLY unanchored choices in panel |

**Anchored choice definition:** A choice is anchored if `(choice.refs ?? []).length > 0`.
Anchored choices are never duplicated in the ActionPanel. This filter applies ONLY for
`type === 'choice'` picks. Element picks are always fully board-anchored.

**Go Fish hybrid example:** When the `ask` action presents `target` (board-anchored, opponent
hand highlighted) and then `rank` (unanchored, text choice), the `rank` step shows in the panel
while the opponent's hand zone is highlighted on the board. These are sequential picks, not
simultaneous — the panel filter runs per-pick.

**Empty panel fallback:** If the unanchored-remainder set is empty (should not happen in a
correctly configured action, but guard defensively), the panel renders an empty state:
"Tap a highlighted element to continue." (14px, weight 400, muted color `#888`).

## Presentation Overlay Contract

> **D-04 / PRESENT-01/02/03:** Per-UI overlay declared as a plain TypeScript object,
> passed as a prop, resolved AFTER engine visibility filtering.

### Overlay Type

```typescript
// src/ui/components/auto-ui/presentation.ts (new file)

export interface PresentationEntry {
  /** URL for the element's face image. Ignored when element.__hidden === true. */
  image?: string;
  /** Override display label. Safe to show on hidden elements (generic, not identity-bearing). */
  label?: string;
  /** Stats block to display. Ignored when element.__hidden === true. */
  stats?: Record<string, unknown>;
  /** Custom Vue component for full render override. Author's responsibility for hidden safety. */
  render?: Component;
}

export interface PresentationOverlay {
  /** Keyed by element.className. Lowest precedence. */
  byClass?: Record<string, PresentationEntry>;
  /** Keyed by element.name. Highest precedence. */
  byName?: Record<string, PresentationEntry>;
  /** Keyed by attribute key → attribute value → entry. Middle precedence. */
  byAttribute?: Record<string, Record<string, PresentationEntry>>;
}
```

### Key Resolution Precedence

```
byName[element.name]              ← highest (instance-level override)
  └─ falls through if no match
byClass[element.className]        ← middle (class-level override)
  └─ falls through if no match
byAttribute[key][value]           ← lowest (attribute-level, first match wins)
  └─ falls through if no match
null                              ← no overlay; renderer uses engine topology defaults
```

**Single resolution per element — no merging.** The first match wins entirely. If `byName`
matches, `byClass` is NOT consulted for fallback fields. Authors needing field-level merging
should flatten their entries explicitly.

### Security Guard — PRESENT-02

`resolvePresentation` MUST apply the following guard:

```typescript
// When element.__hidden === true:
// - SKIP 'image' and 'stats' fields (these could encode identity)
// - ALLOW 'label' (generic text, e.g. "Hidden card" — not identity-bearing)
// - ALLOW 'render' (author's explicit custom component — author's responsibility)
if (element.__hidden) {
  return entry ? { label: entry.label, render: entry.render } : null;
}
```

**Rationale:** Phase 91's `toJSONForPlayer` strips `$images.face` and value-bearing
attributes from hidden elements (lines 2271, 2306, 2326 of `game.ts`). The overlay must
not re-introduce visual cues that encode identity. A card-back URL hardcoded by the author
as `image` would bypass this — hence the guard. The guard operates on `element.__hidden`
from the already-filtered `gameView`.

### Provide/Inject Chain

```
GameShell.vue
  props.presentation?: PresentationOverlay
  provide('presentation', toRef(props, 'presentation'))

AutoUI.vue
  props.presentation?: PresentationOverlay
  passes to :presentation on AutoRenderer

AutoRenderer.vue
  props.presentation?: PresentationOverlay
  provide('presentation', computed(() => props.presentation))

Per-element renderers:
  const overlay = inject<ComputedRef<PresentationOverlay | undefined>>('presentation')
  const entry = computed(() =>
    resolvePresentation(props.element, overlay?.value)
  )
```

### Renderer Slot Mapping

| `PresentationEntry` field | Renderer behavior |
|---------------------------|------------------|
| `image` | `<img :src="entry.image">` replaces the engine's `$image`/`$images.face` default; ignored on `__hidden` elements |
| `label` | Replaces `element.name ?? element.className` in board header, cell notation, card title |
| `stats` | Displayed as a `<dl>` stat block alongside the element; ignored on `__hidden` elements |
| `render` | `<component :is="entry.render" :element="element">` — full override; engine topology not rendered for this element |

**Graceful fallback (no overlay match):** Renderer uses engine topology defaults — bare element
as rendered by Phase 93. No error, no blank space.

### Game-Side Overlay Files

Each game exports its overlay from a sibling file:

```typescript
// ~/BoardSmithGames/{game}/src/ui/presentation.ts
import type { PresentationOverlay } from 'boardsmith/ui';

export const myGameOverlay: PresentationOverlay = {
  byClass: { ... },
  byName: { ... },
};
```

And passes it to `<AutoUI :presentation="myGameOverlay">` or
`<GameShell :presentation="myGameOverlay">`.

**Per-game overlay contents (minimal — Claude's Discretion per CONTEXT.md):**

| Game | Minimal overlay |
|------|-----------------|
| **Hex** | `byClass.HexCell: { label: '' }` to suppress cell name; stones inherit player color from `.player.color` — no image needed |
| **Go Fish** | Cards use existing `$images` — overlay adds `byClass.PlayerHand: { label: 'Hand' }` for hand zone labeling |
| **Checkers** | `byClass.CheckerPiece: { label: '' }` to suppress name; `byClass.Square: { label: '' }` to suppress square name (notation shown as corner label already) |

These are the minimum overlays that make the three games visually clear without custom UI code.
Richer overlays (king images, rank labels) are author's discretion.

## Playability Acceptance Contract

> The playability gate is human-verified browser testing. These are the minimum interaction
> surfaces each game must expose via auto-UI only (no custom UI code in `#game-board` slot).

### Shared: App.vue for all three gates

Each game's `App.vue` must slot ONLY `<AutoUI :presentation="...">` into `#game-board`.
The split-screen custom-UI panel is removed for the gate. This may be temporary or permanent
(per CONTEXT.md "Claude's Discretion") — the planner chooses; this spec requires it for
the gate.

### Hex — Playability Acceptance

| Requirement | Minimum interaction surface |
|-------------|-----------------------------|
| Place a stone | Click any empty hex cell → `HexBoardRenderer` dispatches `triggerElementSelect(cell)` |
| See legal moves | All valid cells show `.action-selectable` state (green pulsing outline) |
| No illegal moves | Non-selectable cells are visually inert; click is no-op |
| Win detection | GameShell flow engine handles; auto-UI shows end-of-game via normal action panel ("Game over" from flow) |
| No custom UI code | `App.vue` `#game-board` slot contains only `<AutoUI>` |

**ActionPanel presence:** Hex `placeStone` uses `chooseElement` (always board-anchored) →
panel is auto-absent after D-02. Footer never shows during play.

### Go Fish — Playability Acceptance

| Requirement | Minimum interaction surface |
|-------------|-----------------------------|
| Ask a player | Click opponent's hand zone (board-anchored `target` choice) → hand zone `.is-board-highlighted` |
| Choose a rank | Rank choices appear in ActionPanel (unanchored `rank` choice → hybrid layout) |
| Draw from deck | Deck click dispatches element selection; or panel button if unanchored |
| Collect books | Automatic flow step; no manual interaction required |
| No custom UI code | `App.vue` `#game-board` slot contains only `<AutoUI>` |

**ActionPanel presence:** Hybrid — `target` pick is board-anchored (panel absent); `rank` pick
is unanchored (panel shows with rank choices). Panel must correctly switch absent/present
between picks.

### Checkers — Playability Acceptance

| Requirement | Minimum interaction surface |
|-------------|-----------------------------|
| Select piece | Click a checker piece → `PieceRenderer` dispatches `triggerElementSelect(piece)` |
| Select destination | Click a destination square → `GridBoardRenderer` dispatches `triggerElementSelect(cell)` |
| Multi-jump highlight | All source + target + intermediate squares highlighted simultaneously via `refs` array |
| King promotion | Visual: overlay `byAttribute.isKing.true: { label: 'K' }` or author choice; interaction is automatic from flow |
| No custom UI code | `App.vue` `#game-board` slot contains only `<AutoUI>` |

**ActionPanel presence:** Both piece selection and destination selection are board-anchored
(`chooseElement` and `chooseFrom` with `boardRefs`) → panel auto-absent throughout play.
