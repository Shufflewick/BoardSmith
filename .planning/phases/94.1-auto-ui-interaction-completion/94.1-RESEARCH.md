## Summary

Phase 94.1 completes the Go Fish and Checkers playability gates deferred from Phase 94.
Three successive fix rounds during Phase 94 passed all controller-unit tests yet left the
browser broken, because the project has **no Vue component / interaction test infrastructure** —
every fix tested the reactive logic in isolation but never exercised the full
controller → bridge → component rendering chain.

The root causes of all three remaining bugs are now identified by code inspection. Bug C
(the deepest) traces to a single line in `ActionPanel.vue` (line 230) that calls a
non-reactive helper function instead of the reactive computed; this prevents the footer
from repopulating choices when async-fetched choices arrive. Bug A (Checkers destination
not selectable) is a downstream consequence: without the D-02 footer change, the board
suppresses the panel AND the bridge doesn't get a second watcher trigger, leaving no
interaction surface. Bug B (Go Fish ask) shares the same non-reactive choices gap — the
'rank' pick auto-populates after 'target' is auto-filled in 2-player games, but the
footer's filtered choices never update.

**Primary recommendation:** Stand up jsdom + `@vue/test-utils` component tests (after
discussing dep addition with user), write tests that reproduce all three bugs against
the real ActionPanel component, then apply the targeted fixes (one-line Bug C fix, D-02
`allCurrentChoicesAnchored` change, 4-5 test assertion updates).

## Bug C Choice-Population Path — Confirmed Root Cause

### The non-reactive gap (verified by reading `ActionPanel.vue:226-255`)

`ActionPanel.vue` computes `filteredChoices` at **line 230**:

```typescript
// src/ui/components/auto-ui/ActionPanel.vue:230
let choices: ChoiceWithRefs[] = actionController.getCurrentChoices() as ChoiceWithRefs[];
```

`getCurrentChoices()` is a bare function (`useActionController.ts:370-373`) that calls
`getChoices(currentPick.value)`. Inside `getChoices`, the choices snapshot is read via
`actionSnapshot.value.pickSnapshots.get(selectionName)` — a `Map.get()` call. **Vue 3
does not track plain Map access.** Only `actionSnapshot.value` itself (a `ref`) is tracked.

`fetchChoicesForPick` (line 530-595) stores choices with `pickSnapshots.set(...)` (Map
mutation — not reactive) then increments `snapshotVersion.value++`. The reactive computed
`currentChoices` (line 647) reads `const _version = snapshotVersion.value` — this is why
the controller computed updates. But `filteredChoices` in ActionPanel calls
`getCurrentChoices()` which **never reads `snapshotVersion`**, so it does NOT re-run when
async choices arrive.

**Fix:** Change `ActionPanel.vue:230` from:
```typescript
let choices = actionController.getCurrentChoices() as ChoiceWithRefs[];
```
to:
```typescript
let choices = (actionController.currentChoices.value as ChoiceWithRefs[]).slice();
```

`currentChoices` is already exported from `useActionController` return value (line 1667 in
`useActionController.ts`). This makes `filteredChoices` reactive to `snapshotVersion`,
so when async choices arrive the footer updates immediately.

### Secondary: filteredValidElements also uses non-reactive helper

`ActionPanel.vue:264`:
```typescript
const validElements = actionController.getValidElements(currentPick.value);
```

`getValidElements(sel)` is non-reactive. For Bug B (Go Fish element picks), if valid
elements are fetched async, the panel won't update. Same fix pattern: use
`actionController.validElements.value` (the reactive computed, line 1651).

### Why controller-unit tests pass but browser fails

The tests in `useActionController.test.ts` (including the "becomes anchored reactively"
test at line 2007) call `controller.fill('piece', 77)` directly and assert on
`controller.currentChoices.value` — the reactive computed. This DOES track `snapshotVersion`
and DOES recompute when choices arrive. The unit test never mounts ActionPanel, so the
non-reactive `getCurrentChoices()` path is never exercised. That is the exact test gap
D-01 addresses.

## Common Pitfalls

### Pitfall 1: Non-reactive choices in ActionPanel
**What goes wrong:** Footer shows "No options available" even when choices were fetched.
**Why it happens:** `filteredChoices` calls `getCurrentChoices()` (non-reactive to
`snapshotVersion`). Choices arrive after the computed first runs; no reactive dep triggers
re-run.
**How to avoid:** Use `actionController.currentChoices.value` in ActionPanel.
**Warning signs:** Controller unit tests pass; mounted component test fails.

### Pitfall 2: `board.clear()` wipes validElements mid-flow
**What goes wrong:** `startAction` in `useBoardActionBridge.ts` (line 181) calls
`board.clear()` AFTER `controller.start()` returns (which includes the async fetch). This
clears `board.validElements` that was just populated.
**Why it happens:** The `watch([currentPick, filteredValidElements, choicesForBoard], ...)`
watcher re-fires when `snapshotVersion` increments (since `choicesForBoard` tracks it).
After `board.clear()`, the watcher fires again and repopulates — but only if `choicesForBoard`
changed after the clear.
**How to avoid:** Component test must `await nextTick()` twice (once for the clear, once
for the re-populate) before asserting board selectability.
**Warning signs:** Board briefly has no selectable cells, then gets them; timing-sensitive.

### Pitfall 3: `board.setCurrentAction` picks stale pick index
**What goes wrong:** In `startAction` (line 182-183), after `controller.start()` pre-fills
the first arg, `currentPick` advances to the SECOND selection. But `board.setCurrentAction`
is called with `firstSel.name` (index 0) — the already-filled selection.
**Why it happens:** `startAction` is designed for the no-pre-fill case; calling it with
`args` is the board-click shortcut path.
**How to avoid:** The bridge watcher immediately corrects via `board.setCurrentPick(idx, name)`
when it fires. But the intermediate state (wrong pick index) can cause visual flicker. A
component test asserting `board.currentPickName` must await the watcher flush.

### Pitfall 4: `filterBy` uses `currentArgs.value` not `collectedPicks`
**What goes wrong:** `getChoices(destination)` applies `filterBy` using
`currentArgs.value['piece']` (line 357 in `useActionController.ts`). If `currentArgs`
is cleared and repopulated, the filter runs correctly. But `buildServerArgs()` uses
`collectedPicks` (not `currentArgs`) — verify both have the same value.
**Why it happens:** Two parallel sources of truth for args values. In the `startAction`
path, `Object.assign(currentArgs.value, initialArgs)` AND `collectedPicks.set('piece', ...)`
are both called (lines 1106 + 1122-1128 in `start()`), so they agree. But if only one is
set, the filter sees different values than the server received.
**Warning signs:** Server returns choices filtered for wrong piece, or client filter
removes all choices.

### Pitfall 5: jsdom SVG limitations
**What goes wrong:** `HexBoardRenderer.vue` uses SVG `<polygon>` elements for hexes.
jsdom renders them but does not compute SVG geometry (no layout engine). Tests using
`getBoundingClientRect()` or SVG transform matrices return zeros.
**Why it happens:** jsdom is a DOM API implementation without a layout engine.
**How to avoid:** Tests for Hex should assert class presence (`isSelectableElement`) and
click dispatch, NOT pixel coordinates or geometry. These tests work in jsdom. Geometry
tests (if ever needed) require a browser environment.
