### INTRO — Action-Space Introspection

#### INTRO-01: `getActionSpace`

**Disposition:** BUILD new method on `Game` (wrapping existing primitives — no parallel validator)

**Exact public name:** `game.getActionSpace(seat)`

**TypeScript signature:**
```typescript
// src/engine/element/game.ts — new public method on Game<P, B>
getActionSpace(seat: number): ActionSpaceView
```

**Return shape:**
```typescript
// New interface — to be added to src/engine/types.ts (or engine public barrel)
interface ActionSpaceView {
  /** Only actions the seat can legally execute right now */
  actions: ActionSchemaView[];
}

interface ActionSchemaView {
  name: string;
  prompt?: string;
  help?: string;
  /** Ordered list of selections for this action */
  selections: PickMetadata[];          // existing type from src/session/types.ts
  /**
   * Ready-to-submit argument template — one key per selection.
   * Placeholder values signal what the caller must fill before submitting.
   *
   * DESIGN DECISION (INTRO-01): arg-template placeholder format
   * - Optional selections  → null
   * - Required selections  → { __required: true }
   * Rationale: null is a natural "no value yet" for optional picks (they are valid
   * to omit). A sentinel object for required picks makes the template self-describing:
   * any tool inspecting the template can identify unfilled required slots without
   * re-reading the PickMetadata array. Flat primitive sentinels (empty string, -1)
   * are ambiguous — a required string selection could legitimately be an empty string.
   */
  argTemplate: Record<string, null | { __required: true }>;
}
```

**Implementation constraint:** `getActionSpace` MUST be implemented as:
1. `availableActionsForSeat(game.getFlowState(), seat)` → `string[]`
2. resolve `Player` object for `seat`
3. `buildActionMetadata(game, player, availableActionNames)` → `Record<string, ActionMetadata>`
4. Enrich each `ActionMetadata` entry with the `argTemplate` field

No parallel condition evaluation. Do not call `evaluateCondition()` directly from this method.

**Serialization format:** `ActionSpaceView` is a plain JSON-serializable object. All `PickMetadata` nested values are already plain types (no element references in the template — those appear in the submitted args, not the template itself). Safe to `JSON.stringify` for WebSocket transmission.

**Owning module:** engine (`src/engine/element/game.ts`)

#### INTRO-03: `buildActionArgs`

**Disposition:** BUILD new public utility function

**Exact public name:** `buildActionArgs(actionName, selectionValues, game, seat)`

**TypeScript signature:**
```typescript
// src/engine/utils/arg-builder.ts — new file
function buildActionArgs(
  actionName: string,
  selectionValues: Record<string, unknown>,   // plain JS values: element objects, strings, numbers
  game: Game,
  seat: number,
  options?: { format?: 'in-process' | 'wire' }  // default: 'in-process'
): Record<string, unknown>
```

**DESIGN DECISION (INTRO-03): wire-format vs in-process args**

The `runner.performAction(actionName, playerIndex, args)` call accepts element objects directly
for in-process use (it serializes internally). The `{ __elementRef: string }` wire format is
only required for cross-process transmission (WebSocket/agent-to-server).

- **`'in-process'` (default):** Returns plain element-object args suitable for `runner.performAction` and `testGame.performAction`. No serialization applied to element arguments. Suitable for: TestGame, headless AI, `enumerateLegalMoves`.
- **`'wire'` format:** Returns serialized args where element objects are replaced with `{ __elementRef: string }` (branch path) or `{ __elementId: number }`. Suitable for: cross-process agents sending actions over WebSocket.

Rationale: The default should serve the 90% use case (headless testing and AI). Cross-process agents opt in explicitly via `{ format: 'wire' }`.

**Return shape:** `Record<string, unknown>` — shape depends on format option as described above.

**Serialization format:** In-process return is NOT JSON-safe (may contain element objects). Wire-format return IS JSON-safe.

**Owning module:** engine (`src/engine/utils/arg-builder.ts`)


#### PIT-01: `loop()` construction-time `maxIterations` enforcement

**Disposition:** BUILD (breaking change) — replace `devWarn` path with unconditional construction-time throw

**Location:** `src/engine/flow/builders.ts:85–96`

```typescript
if (config.maxIterations === undefined) {
```

```typescript
if (config.maxIterations === undefined) {
```

**Note:** The existing runtime iteration cap at `src/engine/flow/engine.ts:1045` (`DEFAULT_MAX_ITERATIONS = 10000`) remains as a final safety net. It is now secondary — the construction-time throw catches the common omission case.

#### PIT-02: Element class registration validation at `startFlow()`

**Disposition:** BUILD — new validation pass during the first `startFlow()` flow-tree traversal

**Validation scope (RESOLVED — per Open Question 4):** Classes encountered during the first `startFlow()` traversal only. Post-construction dynamic queries are explicitly out of scope and documented as a known limitation.

**Implementation point:** `src/engine/element/game.ts:1426` (`game.startFlow()`)

**Validation logic:**
1. During the first flow traversal, collect every element class accessed via `game.all(SomeClass)`, `game.firstOf(SomeClass)`, or similar class-query methods
2. Cross-check each collected class against `game.classRegistry`
3. Throw on the first unregistered class encountered:

```typescript
throw new Error(
  `Element class '${className}' is used in the game flow but was not registered. ` +
  `Call game.registerElements([${className}]) in your game constructor before calling startFlow().`
);
```

**Known limitation (document in public docs):** Element classes accessed in async event handlers or code that runs after `startFlow()` completes cannot be pre-validated. This is an accepted scope boundary — no static analysis is performed.

**Owning module:** engine (`src/engine/element/game.ts`)

#### PIT-03: Action reachability validation at `startFlow()`

**Disposition:** BUILD — new flow-tree traversal at `startFlow()` time

**Implementation point:** `src/engine/element/game.ts:1426` (`game.startFlow()`)

**Validation logic:**
1. Enumerate all `actionStep` nodes in the flow tree (recursive traversal of the stored flow node structure via `game._flow`)
2. Collect the union of all action names referenced across all `actionStep({ actions: [...] })` nodes
3. For each collected name, verify `game.getAction(actionName)` returns a defined action
4. Throw on any `actionStep`-referenced name that is not registered:

```typescript
throw new Error(
  `Action '${actionName}' is referenced in an actionStep but was not registered. ` +
  `Call game.registerAction('${actionName}', { ... }) or game.registerActions([...]) ` +
  `in your game constructor before calling startFlow().`
);
```

**Bidirectional check (devWarn, not throw):** If a registered action is NOT referenced in any `actionStep` node, emit a `devWarn` — the action is registered but unreachable from any flow node, which is suspicious but may be intentional (e.g., actions invoked via non-actionStep paths).

**Owning module:** engine (`src/engine/element/game.ts`)
