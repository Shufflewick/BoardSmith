Methodology & how to read this

Three dimensions were each swept by dedicated finder agents that read real source (Read/Grep/Bash), ran the repo's own tooling (fallow dead-code, jscpd duplication) where relevant, and were required to cite file:line evidence. Each finding then went to an independent skeptic verifier instructed to assume the finding is wrong until the code proves otherwise, open the cited files, and assign its own severity. Where the verifier disagreed, the badge shows the verifier's rating and a note records the original. Rejected claims are listed separately at the bottom.

  • Dead Code = unused exports/files, unreachable branches, dead options, orphaned modules, duplication.
  • Docs ↔ Code = documented APIs/behavior/examples that no longer match the actual source.
  • Pit-of-Success = places the easy path is the wrong path — footguns, silent failures, invalid states representable in types.
  • Severity = how badly it undermines correctness or the pit-of-success goal · Ease of fix = rough cost (easy/moderate/hard) · Confidence = finder's confidence it's real.

Per-dimension assessment

Dead Code 16 confirmed

The Dead Code dimension is broadly healthy in its essentials but carries a meaningful tail of accumulated cruft, with most findings rated low severity and the codebase showing no evidence of dead logic in critical runtime paths. The clearest signal is post-refactor residue: an orphaned `better-sqlite3` dependency and a vestigial RNG call-count tracker left behind after the server/SQLite-store removal, plus fully unused exported modules (`replay.ts`, the testing `fixtures.ts` ScenarioBuilder suite, and a cluster of designer helpers, debug/assertion exports, and the `useHoverHighlight` composable) that appear to be speculative or abandoned API surface with zero consumers. More concerning are a few medium-severity items that risk active confusion rather than mere bloat: a dead, divergent duplicate of `generateCandidateFeatures()` sharing an identical name but different logic, ~85 lines of verbatim-duplicated lobby session callbacks across `GameSession.create`/`restore`, game-option metadata types duplicated as two sources of truth across session and types, and an orphaned `ParallelTrainer` subtree documented as the primary training API yet driven by no command or consumer. Smaller hygiene issues round it out — no-op stubs (`saveAICode()`), dead private helpers, a self-importing barrel that creates circular dependencies, and CLI flags (`validate --fix/--skip-simulation`, `lint --fix`) accepted but silently ignored. None of these block functionality, but the divergent-duplicate and silently-ignored-flag findings undercut the "pit of success" principle and should be prioritized over the larger volume of simple unused-export removals.

Docs ↔ Code 14 confirmed

The Docs-vs-Code dimension is the weakest area surfaced by the audit: documentation has drifted substantially from the actual source API, with five high-severity cases where docs describe interfaces that simply do not exist — a fabricated `game.players` collection (`.get()`, `.current`, `.all()`, `.nextAfter()`), a fabricated GameConnection API in client.md (~12 nonexistent methods), a custom-Player constructor signature the base class lacks, ui-components.md's `actionController.error`/`.pendingAction` (real names: `lastError`/`currentAction`), and an ai-system.md GameSession example invoking a private constructor, wrong option names, and a nonexistent `start()`. Medium-severity drift compounds this across runtime.md (GameRunner's `executeAction`/`getActionHistory`/`getStateAt` instead of the real positional-arg `performAction`), dice-and-scoring.md (UniqueTrack `pointsPerEntry` shown as a function but typed as `number[]`), composable return members (`getAlgebraic`/`pixelToCell`, `hexToPixel`/`pixelToHex`), and architecture.md still documenting the deleted `server` and `worker` modules. A scatter of low-severity naming mismatches (ClaimPositionRequest vs. ClaimSeatRequest, importing `Game` from `boardsmith/ai`) rounds out the pattern. The encouraging signal is that nearly every finding is a documentation-accuracy fix rather than a code defect — the underlying APIs are real and coherent, so the corrective work is mechanical reconciliation against source rather than redesign; still, the breadth and the number of high-severity fabrications mean the docs are currently a liability for new developers and should be treated as a blocking cleanup, ideally backed by doc-example compilation tests to prevent recurrence.

Pit-of-Success 22 confirmed

The Pit-of-Success dimension shows systemic erosion: the codebase repeatedly makes the wrong path the easy path and the right path the deliberate one. Security and correctness default the wrong way — Deck and Hand expose their contents unless the designer remembers an extra opt-in call, chooseElements silently drops invalid IDs and never enforces multiSelect min/max, and the Client SDK mints player-identity tokens with Math.random(); meanwhile the state layer harbors high-severity recording gaps where completed pending/repeating actions never reach actionHistory and undo computes turn boundaries with a heuristic the code itself flags as broken, so replay, undo counts, and the canUndo display silently disagree. A strong "fail silent" anti-pattern runs throughout: loop()'s maxIterations completes instead of erroring, onSelect and simulateRandomGames swallow errors while still committing or reporting success, random simulation emits a seed that can't reproduce its own failure, and documented-but-inert features ('unordered' visibility, triggerChoiceSelect, registerBeforeAutoExecute, useDragDrop outside a provider) present capabilities that quietly do nothing — each a trap that hides bugs rather than surfacing them. APIs also overstate their generality (useDragDrop auto-wires only two hard-coded shapes; WebSocketMessage is a stringly-typed all-optional bag) and lose structured information already in hand, as when GameSession re-derives errorCode by substring-matching such that NOT_YOUR_TURN is never produced. Not everything is broken — useBoardInteraction correctly throws outside its provider (a good fail-loud model its drag-drop sibling should follow), the runtime honestly documents action-replay restore as unsound, and structured error codes exist at the runner layer — but these healthy patterns are inconsistently applied, leaving the overall surface tilted toward quiet failure and insecure defaults that demand vigilance rather than rewarding it.
