Methodology & how to read this

Each of the five dimensions was assigned a dedicated adversarial judge that read real source (Read/Grep/Bash), and was required to cite file:line evidence for every claim. Each finding then went to an independent skeptic verifier instructed to assume the finding is wrong until the code proves otherwise, open the cited files, and assign its own severity. Where the verifier disagreed with the judge, the badge shows the verifier's rating and a note records the original. The two findings the verifier rejected are listed separately at the bottom — kept for transparency, not counted in the ranking.

  • Severity = how badly it undermines the pit-of-success goal (trap, silent failure, footgun, data-loss, or security hazard).
  • Ease of fix = rough implementation cost (easy / moderate / hard).
  • Confidence = the judge's confidence the issue is real and impactful.

Per-dimension assessment

Architecture 7 confirmed

BoardSmith's engine is broadly thoughtful — the recent move to a state-authoritative `GameRunner.fromSnapshot` (runner.ts:303) is a genuinely good architectural decision, command/element/flow separation is mostly clean, and error messages are generally actionable. But the persistence/restore layer has accreted into a sprawl of overlapping, partially-contradictory mechanisms that violate single-source-of-truth and, worse, disagree about correctness: the same "rebuild a game" operation is implemented five different ways with three different soundness guarantees, and the debug/time-travel path still uses the very replay mechanism the rest of the system was rewritten to abandon as "unsound." On top of that, the `Game` class is a 2684-LOC god object, declared array state is silently swapped for dev-only Proxies (creating dual sources of truth and dev/prod snapshot divergence), the cross-cutting "whose turn" invariant is represented two ways in nine files, and a shared selection-state-machine type carries a `Set` that silently dies when it crosses the stateless executor boundary. These are exactly the traps that bite a designer or AI agent who trusts the abstraction.

Security 8 confirmed

The BoardSmith engine's per-player state filtering (getPlayerView) and the lobby's host-only logic checks are individually well-written, but the network layer that exposes them has essentially no authentication or authorization model. Player identity is entirely self-asserted: every HTTP handler trusts a `player`/`playerId` field supplied directly in the request, and the Cloudflare worker's WebSocket upgrade derives the player seat from unverified URL query params. As a result, the careful hidden-information filtering and "host only" lobby gates are trivially bypassable over the wire — any client can read another player's private state, act as any player, impersonate the host, and (via always-on debug routes) stack/peek decks, rewind, or wipe games. Separately, the "sandbox" advertised in the worker docs does not exist at runtime: the only runtime guard (validateCode/withLimits) is never invoked, so game code runs with full host privileges and the eslint rules are lint-time only. These are systemic Pit-of-Failure traps: a designer or AI agent who ships a hidden-information game on the provided server/worker will silently ship a cheatable, data-leaking game while believing the framework handled it. The strongest fixes are to (1) bind identity to a server-issued token/session and (2) make every handler derive `player` from that bound identity rather than from the request body/query.

Performance 3 confirmed

Incremental designs are mostly sound; the undo/checkpoint subsystem and MCTS are the main performance liabilities (3 high-severity findings below).

Usability 6 confirmed

BoardSmith's authoring surface is largely thoughtful — it has guard rails (devWarn for loop safety, ElementCollection.includes/indexOf warnings steering to contains(), DEFAULT_MAX_ITERATIONS runtime cap so loops can't truly hang, an unknown-action typo warning in the flow engine, and many "PIT OF SUCCESS"-labeled auto-unwrap/auto-fill behaviors in useActionController). The ActionPanel uses native <button> elements throughout, which gives keyboard/focus accessibility for free. However, there are genuine footguns that undermine the Pit-of-Success goal for both human designers and AI agents: the single most-used authoring API (the action builder's execute handler) provides zero type linkage from declared selections to handler args, forcing manual `as` casts on every game; a prominent error message points users to a method that does not exist (`game.defineAction`); the board-interaction composable that a HARD RULE depends on fails silently when missing; and the element-matching logic uses OR-semantics across id/name/notation that can select the wrong element. There is also confusing overlap between three element-selection builder methods. None of these are catastrophic data-loss bugs, but several are classic traps where the easy path silently does the wrong thing.

DX / Agent-XP 5 confirmed

BoardSmith has genuinely strong loud-failure ergonomics in several places (the `fromJSON` "Unknown element class" error, the flow infinite-loop guard with a stack trace, the `dependentFilter` crash message, and `boardsmith validate`'s determinism scan are all excellent, actionable, pit-of-success error messages). The getting-started doc is accurate and the action/flow builder APIs are mostly discoverable. However, the single most important audience — an AI agent turning a one-line idea into a working game — is undermined by one structural type-safety hole and a cluster of toolchain/doc inconsistencies. The fluent action builder throws away all type information: `ctx.game` is `Game` (defaulted to `any`) and every `args.*` is `unknown`, so the primary API an agent writes (action handlers) is a wall of unchecked `as` casts in every real example game — a silent-mismatch trap. The sandbox guardrails (the eslint-plugin) are dead code never wired into scaffolded projects (nor dogfooded by the repo), with their enforcement re-implemented three separate, inconsistent times. And wrong package import paths (`@boardsmith/engine`, `@boardsmith/ai`, etc.) are baked into runtime "fix-it" error messages, the AI code generator's output, and the slash-command templates an agent follows — so generated code won't resolve and the suggested fixes mislead. These are concrete, fixable issues that directly steer agents into traps.

HTTP action endpoint trusts client-supplied player identity (no authn) — act as any player

judge said critical, verifier set high
Blocked: a correct fix needs a design decision (see verifier note & the auth-token gap), so it was intentionally not merged.
Why it's a concern
There is no binding between the network caller and the `player` they claim to be. A `curl -X POST /games/<id>/action -d '{"action":"...","player":2,"args":{...}}'` submits moves on behalf of player 2. The WebSocket path correctly uses the server-tracked `session.playerSeat` (core.ts:505-509), but the HTTP route — shipped in both the standalone server and the production worker — bypasses that entirely.
Pit-of-Success impact
A designer/AI agent who builds a game and deploys it on the provided server or worker gets an authoritative-looking server that is actually trivially cheatable: opponents can play your turns, force passes, or grief. The framework gives the impression of enforcing turns/seats while the easy HTTP path silently ignores identity. Wrong usage is the default.
Path(s) to fix  (moderate)
src/server/handlers/games.ts (handleAction), src/worker/index.ts (#handleAction). Derive `player` from an authenticated session/token bound at connect/create time (e.g. a per-seat secret returned from claim-seat/join), not from request.player. Reject if the bound identity != requested player.

Action handlers are completely untyped: ctx.game is Game<any> and every arg is unknown, forcing unchecked casts everywhere

judge said critical, verifier set medium
Why it's a concern
Casts (`as`) are unchecked assertions. If an agent renames a selection, changes its type, or typos an arg name, TypeScript stays silent and the bug surfaces only at runtime (or as `undefined` propagating into game logic). The library's own motto is 'impossible to represent invalid states in types' — here the opposite is true: the type system actively certifies wrong code as correct.
Pit-of-Success impact
This is the deepest trap for AI agents. Agents rely heavily on the type checker as a correctness oracle while iterating; here it provides none for the core game-logic surface. An agent writing `args.card as Card` when the selection is actually named `cards` (plural) gets a clean compile and a runtime `undefined`. Every example game models the cast-everything pattern, so agents are actively trained to bypass type safety.
Path(s) to fix  (hard)
Make `ActionContext` generic (`ActionContext<G extends Game = Game>`) and make the `Action`/builder generic over an accumulating args record so `chooseFrom<T>('card', ...)` adds `{ card: T }` to the execute args type. Thread the game type from `Game<G>` into `registerAction`/`Action.create<G>()`. At minimum, type `args` per-selection so `execute(({ card }) => ...)` infers `card`'s type. Files: src/engine/action/types.ts, src/engine/action/action-builder.ts, src/engine/element/game.ts (registerAction signature).
